# PyCat v0.0.6

> Release notes for GitHub Release

## Suggested release title

`PyCat v0.0.6`

## Suggested tag

`v0.0.6`

## Release body

PyCat v0.0.6 聚焦 **Windows Nuitka 发布体系、GUI/CLI 入口解耦、python_exec 打包运行稳定性、工具结果实时刷新** 四个方向。该版本不追求向历史入口兼容，而是进一步收敛架构边界：`pycat.exe` 明确作为 GUI 入口，`pycat-cli.exe` 明确作为控制台入口，核心运行时由 GUI 与 CLI 共享。

### 🧭 本版本定位

v0.0.6 是一个偏工程化和发布质量的版本，重点解决早期打包后暴露出的真实 Windows 使用问题：

- GUI 程序双击启动不应弹出 CMD 窗口。
- CLI 命令在 PowerShell / CMD 中应具有正常的同步输出行为。
- Nuitka GUI 子系统下的 `python_exec` 工具应能稳定执行临时 Python 脚本。
- 工具执行结果应在 UI 中实时刷新，而不是依赖重新打开会话。
- 入口文件职责应清晰，不再由一个入口同时承担 GUI 与 CLI 分发。

## ✨ Highlights

- **GUI / CLI 双入口发布**：`pycat.exe` 作为无控制台 GUI 入口，`pycat-cli.exe` 作为标准控制台 CLI 入口，解决单 exe 在 Windows 上难以同时满足双击体验和终端输出语义的问题。
- **入口架构解耦**：`main.py` 回归为纯 GUI 入口，仅保留 `python_exec` worker 分支；`main_cli.py` 成为独立 CLI 入口；CLI 不再从 GUI 入口动态分流。
- **Nuitka 构建优化**：构建脚本拆分 GUI 与 CLI 子系统配置。GUI 使用 `--windows-console-mode=disable`，CLI 使用 `--windows-console-mode=force`，最终发布目录包含 `pycat.exe` 与 `pycat-cli.exe`。
- **CLI 初版能力落地**：新增 headless CLI，支持 `run`、`chat`、`list`、`version`，便于脚本化调用、自动化验证和后续 TUI/服务化扩展。
- **python_exec 打包运行修复**：Nuitka 打包后，`python_exec` 使用当前 exe 的隐藏 worker 执行临时脚本，不依赖系统额外安装 Python；同时修复 GUI 子系统下继承无效标准句柄导致的 `[WinError 6] 句柄无效`。
- **工具结果 UI 实时刷新**：工具结果事件改为瞬态 `Message(role="tool")`，UI presenter 将其合并到对应 `assistant.tool_calls[].result` 并刷新已有工具调用卡片，避免结果只有重新打开会话后才显示。

## 🔧 从 v0.0.5 以来的主要变更

### 新增

- **CLI 包**：新增 `cli/`，包含命令解析、运行时适配和终端输出适配。
- **CLI 入口**：新增 `main_cli.py`，作为 `pycat-cli.exe` 的唯一打包入口。
- **python_exec worker**：新增 `core/tools/system/python_worker.py`，为 Nuitka 打包环境提供隐藏脚本执行入口。
- **Nuitka CLI smoke test**：构建脚本在打包后自动验证 `pycat-cli.exe --version` 与 `pycat-cli.exe list modes`。
- **worker smoke test**：构建脚本新增 `python_exec` worker 探针，验证打包后 GUI exe 可执行临时 Python 脚本并写入 marker。

### 优化

- **入口职责简化**：`main.py` 不再导入或分发 CLI，只负责 GUI 启动和 worker 分支。
- **发布产物语义清晰化**：
  - `pycat.exe`：桌面 GUI 应用，无控制台弹框。
  - `pycat-cli.exe`：命令行入口，适合 PowerShell / CMD / 自动化脚本。
- **构建脚本分层**：GUI 和 CLI 分别使用不同 Nuitka console mode，避免 `attach` 模式下 CMD prompt 与输出顺序错乱。
- **CLI 模型选择优先级**：CLI 支持按 `CLI 参数 > 环境变量 > 应用设置/默认模型` 的顺序解析 provider/model。
- **CLI 工具审批策略**：`pycat-cli run` 默认自动批准工具调用，便于非交互式使用。
- **工具结果持久化路径统一**：继续保持顶层 transcript 只保存 `system/user/assistant`，工具结果只写入 `assistant.tool_calls[].result`。

### 修复

- 修复 `pycat.exe --version` 在 GUI 子系统/attach 模式下无输出或输出顺序异常的问题，改由 `pycat-cli.exe --version` 承担 CLI 语义。
- 修复 Nuitka 打包后 `python_exec` 报 `Python execution error: [WinError 6] 句柄无效` 的问题。
- 修复 GUI 中工具调用已完成但输出不实时更新的问题。
- 修复 `python_exec` 在打包环境中依赖外部 `python.exe` 的问题。
- 修复 CLI 输出流在测试重定向场景下绑定过早的问题。
- 修复 GUI 子系统 exe 在 PowerShell smoke test 中直接调用时等待行为不可靠的问题，worker 探针改用 `Start-Process -Wait -PassThru`。

## 🧱 架构说明

### 入口拆分

```text
main.py          # GUI 入口：构建为 pycat.exe
main_cli.py      # CLI 入口：构建为 pycat-cli.exe
cli/             # CLI 命令层
ui/              # GUI 表现层
core/            # 共享任务运行时、工具、模式、状态与提示词能力
services/        # 应用服务层
models/          # 数据模型
```

这种拆分避免了一个入口同时处理 GUI、CLI、worker 的职责混杂。当前规则是：

- GUI 走 `main.py` / `pycat.exe`。
- CLI 走 `main_cli.py` / `pycat-cli.exe`。
- `python_exec` worker 仍由 GUI exe 内部隐藏分支承担，以保证打包环境下无需外部 Python。

### RunTree / 工具结果规则

本版本继续强化运行树规则：

- 顶层对话只持久化 `system`、`user`、`assistant`。
- `role="tool"` 只作为运行时瞬态事件存在。
- 工具结果只持久化到 `assistant.tool_calls[].result`。
- UI 收到工具结果事件后刷新已有 assistant 工具调用卡片，而不是追加顶层工具消息。

## 🚀 How to use

1. 下载 `pycat-0.0.6-windows-x64.zip`。
2. 解压到本地目录。
3. 双击运行 `pycat.exe` 启动 GUI。
4. 在终端中运行 CLI：

```text
pycat-cli.exe --version
pycat-cli.exe list modes
pycat-cli.exe list providers
pycat-cli.exe list tools
pycat-cli.exe run "你好，介绍一下当前能力"
```

源码开发环境中：

```text
python main.py          # 启动 GUI
python main_cli.py --version
python main_cli.py list modes
```

## 📝 Notes

- 从 v0.0.6 开始，`main.py` 不再承担 CLI 分发；源码下 CLI 请使用 `python main_cli.py ...`。
- `pycat.exe` 是 GUI 子系统程序，设计目标是双击无控制台弹框；请不要再用它承担 CLI 输出语义。
- `pycat-cli.exe` 是控制台子系统程序，适合脚本、终端、CI、自动化任务。
- 当前 Windows 资产未进行代码签名，首次运行如出现 SmartScreen 提示属于常见现象。
- CLI 初版仍以复用现有核心运行时为主，后续可继续演进 TUI、JSON streaming、批处理任务和更多自动化入口。
