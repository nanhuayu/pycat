# PyCat v0.0.2

> Release notes for GitHub Release

## Suggested release title

`PyCat v0.0.2`

## Suggested tag

`v0.0.2`

## Release body

PyCat v0.0.2 在 v0.0.1 的 Chat / Agent / Tools 桌面一体化基础上，集中完善了**多平台 IM 频道接入**能力，并对构建产物、协议依赖和运行时稳定性做了重要优化。

### 🧭 项目定位

在 Claude Code（终端编码 Agent，Rust/Shell/TS）、Codex（终端编码 Agent，Rust 96%）、Cherry Studio（Electron 桌面客户端，TS + Electron ~200MB）、Chatbox（Electron 桌面客户端，TS + Electron ~200MB）、OpenClaw（终端编码 Agent）等成熟项目之间，**PyCat 选择了一条不同的技术路线**：

| 维度 | 主流方案 | PyCat 方案 |
|------|---------|------------|
| **语言** | TypeScript + Rust + Electron + Bun（3-5 种） | **Python 单语言** |
| **桌面 UI** | Electron（捆绑 Chromium + Node.js，~200MB+） | **PyQt6 原生控件（Nuitka 编译，~80MB）** |
| **编译产物** | ASAR 打包 + 解释执行 | **Nuitka 原生机器码编译** |
| **IM 频道** | 无 | **QQ / 微信 / 飞书 / Telegram 四平台** |
| **扩展方式** | IPC / SDK / 插件子系统 | **直接 Python import** |

**PyCat 的技术特点：**

1. **Python 单一技术栈** — 不需要在多种语言间切换，降低上手和维护门槛。
2. **Nuitka 原生编译，包体积更小** — 不捆绑浏览器内核，~80MB 即可独立运行。
3. **四大 IM 频道内置** — QQ、微信、飞书、Telegram 均已接入基础功能。
4. **原生 Python 可扩展** — Skills、Tools、MCP、Channels 全部在 Python 中直接实现。
5. **架构轻量，迭代迅速** — 从 v0.0.1 至今已完成四大频道落地、架构分包、SDK 轻量化。

> ⚠️ **诚实地说明**：PyCat 目前处于早期阶段，功能深度、稳定性、社区规模远不及上述成熟项目。Cherry Studio 在可扩展性和响应格式覆盖上更完善，Chatbox 在移动端体验上更成熟。如果你的目标是找一个生产就绪的桌面 AI 工具，建议优先考虑它们。如果你对 Python + Nuitka + IM 频道这条技术路线感兴趣，欢迎来一起完善。

## ✨ Highlights

- **四平台 IM 频道接入**：微信（二维码桥接）、QQ Bot（官方 Gateway 长连接）、飞书（长连接）、Telegram（Bot API 长轮询）四种主流 IM 已全部通过 `core/channel/sources/<source>/` 分包落地。
- **飞书 WebSocket 轻量化**：移除 `lark-oapi` 大 SDK 依赖，改用本地最小 protobuf wire codec，显著减少 Nuitka 构建时间和产物体积。
- **构建产物精简**：可执行文件统一为 `pycat.exe`，发布压缩包按版本命名（如 `pycat-0.0.2-windows-x64.zip`），便于版本管理和分发。
- **QQ Bot 官方 Gateway**：按 OpenClaw 官方实现接入 QQ Bot Gateway 长连接，只需填写 App ID 与 App Secret，自动完成 Hello、Identify、Heartbeat 与消息回发目标绑定。
- **Telegram Bot API 长轮询**：通过 `getUpdates` 长轮询接收入站消息，无需公网 webhook；`Chat ID` 改为可选，真实对话优先从入站 update 自动绑定。
- **WeChat QR bridge 超时降噪**：长轮询中的临时网络超时不再误报为通道失效，避免扫码凭据被意外清空。
- **模式入口收敛**：主模式固定为 `chat / agent / plan / explore`，并修复未知模式 slug 被强制回退为 `chat` 的问题。
- **Thinking / Reasoning 历史回放修复**：处理了思考模型历史中缺乏 reasoning 字段导致的 `invalid_request_error`，避免脏 assistant 历史滚雪球。
- **Channel 流式事件可见性**：频道触发的工具调用现在可在桌面端实时看到工具步骤和中间输出，不再等到整轮结束。

## 🔧 从 v0.0.1 以来的主要变更

### 新增

- **QQ Bot 官方 Gateway 长连接**：`core/channel/sources/qqbot/` 整套 source 实现
- **Telegram Bot API 长轮询**：`core/channel/sources/telegram/` 整套 source 实现
- **飞书 WebSocket 轻量协议**：`core/channel/sources/feishu/ws_protocol.py` 本地 protobuf codec
- **Channel source 分包架构**：`core/channel/sources/<source>/` 作为平台实现唯一位置
- **ChannelRuntimeContext**：source backend / transport 调用 runtime 的唯一门面
- **Explore 模式**：只读探索、代码定位和架构问答
- **Feishu 协议专项测试**：`tests/test_feishu_ws_protocol.py`（6 tests）
- **飞书 WS 分片重组测试**：已纳入 `test_channel_runtime.py`

### 优化

- 构建产物重命名为 `pycat.exe`，zip 带版本号
- 移除 `lark-oapi` SDK 依赖（Nuitka 构建体积和时间显著降低）
- QQ Bot 默认连接方式改为 websocket，只要求 App ID + App Secret
- WeChat QR bridge 长轮询临时网络超时只记 debug，不触发重新扫码
- 未知 mode slug 保留自身标识，不再强制改写为 `chat`
- 频道设置页精简为微信 / QQBot / 飞书 / Telegram 四个主频道
- 右侧辅助面板改为可折叠分组，避免堆叠过长

### 修复

- QQ Bot "尚无可用目标，无法回发消息"（缺少入站 → binding 链路）
- 思考模型历史 reasoning 字段缺失导致 400 错误
- Channel 工具调用在桌面端无法实时显示中间步骤
- `ChannelConversationBindingStore` 测试与真实数据目录隔离
